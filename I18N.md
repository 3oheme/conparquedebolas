# i18n — Internacionalización (ES + EN)

## Decisión

- **Idiomas:** español (por defecto) e inglés. No se añaden más en esta fase.
- **URL:** español en raíz (`/`), inglés con prefijo `/en/`.
- **Datos:** un único YAML por provincia con campos sufijados `_en` para lo traducible. NO se duplican ficheros.
- **Política ante traducciones faltantes:** si un restaurante no tiene su variante `_en`, NO se publica en `/en/`. Nunca mezclar idiomas en una misma página.

---

## Estructura de URLs

| Idioma | Home | Ciudad | Restaurante | Filtro por tag | Filtro ciudad + tag |
|---|---|---|---|---|---|
| ES | `/` | `/ciudades/[slug]/` | `/restaurantes/[slug]/` | `/con/[tag]/` | `/ciudades/[slug]/con/[tag]/` |
| EN | `/en/` | `/en/ciudades/[slug]/` | `/en/restaurantes/[slug]/` | `/en/con/[tag]/` | `/en/ciudades/[slug]/con/[tag]/` |

Reglas:
- Los `slug` de ciudades y restaurantes son **idénticos** en ambos idiomas. No se traducen.
- Los nombres de ciudades en URLs permanecen en español (`jerez-de-la-frontera`, no `jerez`).
- Los `slug` de los tags también permanecen en español en ambos idiomas (`/en/con/zona-juegos/`, no `/en/con/play-area/`). Las etiquetas visibles en pantalla sí se traducen vía `i18n.yml`.
- Las páginas `/en/con/[tag]/` y `/en/ciudades/[slug]/con/[tag]/` solo se generan si contienen al menos un restaurante con traducción EN. Si no hay ninguno, no se publica la página.

---

## YAML — campos traducibles y no traducibles

Estructura tras añadir i18n:

```yaml
- nombre: "El Faro de Cádiz"                   # NO se traduce
  slug: "el-faro-de-cadiz"                     # NO se traduce
  ciudad: "Cádiz"                              # NO se traduce
  direccion: "Calle Ejemplo, 1, 11401 Cádiz"   # NO se traduce

  por_que_kids_friendly: >
    Personal con paciencia infinita, terraza grande con sombra y juguetes.
  por_que_kids_friendly_en: >
    Endlessly patient staff, large shaded terrace with toys.

  descripcion_general: "Cocina andaluza. Precio medio."
  descripcion_general_en: "Andalusian cuisine. Mid-range."

  # Sin cambios respecto al esquema actual:
  valoracion_kids: 4.5
  imagen: "/assets/img/restaurantes/el-faro-de-cadiz.jpg"
  web: "https://www.ejemplo.com"
  instagram: "el_faro_cadiz"
  lat: 36.5268
  lng: -6.2989
  ultima_verificacion: "2025-04-30"
  precio: "€€"
  tags:
    tronas: true
    carta_infantil: true
    # ...
```

Tabla de referencia:

| Campo | Acción |
|---|---|
| `nombre`, `slug`, `ciudad`, `direccion` | NO traducir |
| `por_que_kids_friendly` | Traducir (campo principal) |
| `descripcion_general` | Traducir |
| `web`, `instagram`, `imagen` | NO traducir |
| `valoracion_kids`, `lat`, `lng`, `precio`, `tags`, `ultima_verificacion` | No aplica |

## Cambios en `_plugins/restaurantes_generator.rb`

El generador actual produce una página por restaurante, una por ciudad, una por tag y una por ciudad+tag. Tras los cambios debe producir **dos versiones** de cada una (ES y EN):

1. Para cada restaurante en el YAML:
   - Generar siempre la versión ES en `/restaurantes/[slug]/` con `lang: "es"`.
   - Generar la versión EN en `/en/restaurantes/[slug]/` con `lang: "en"` **solo si** `por_que_kids_friendly_en` existe y no está vacío. Si falta, omitir.
   - Pasar `has_en: true/false` a cada página ES para que el switcher sepa si debe mostrarse.

2. Para cada ciudad:
   - Generar siempre `/ciudades/[slug]/` con `lang: "es"`.
   - Generar `/en/ciudades/[slug]/` con `lang: "en"` listando únicamente los restaurantes que tengan traducción EN. Si la ciudad acaba sin restaurantes en inglés, no generar la página EN de esa ciudad.
   - Pasar `has_en: true/false` a la página ES de ciudad.

3. Para cada tag:
   - Generar `/con/[tag-slug]/` con `lang: "es"` con todos los restaurantes que tienen ese tag.
   - Generar `/en/con/[tag-slug]/` con `lang: "en"` con los restaurantes que tienen ese tag **y** tienen traducción EN. Omitir si quedan cero restaurantes.
   - Pasar `has_en: true/false` a la página ES del tag.

4. Para cada combinación ciudad + tag:
   - Igual que el punto anterior pero dentro de `/ciudades/[slug]/con/[tag-slug]/` y `/en/ciudades/[slug]/con/[tag-slug]/`.

5. Pasar al layout estas variables de página:
   - `lang` (`"es"` o `"en"`)
   - `slug`
   - `has_en` (booleano — solo en páginas ES, para el switcher)
   - Para restaurantes: el restaurante completo
   - Para ciudades y filtros: la lista filtrada de restaurantes en ese idioma

## Cambios en layouts (`_layouts/restaurante.html`, `_layouts/ciudad.html`, `_layouts/filtro.html`)

1. Leer `page.lang` al inicio del layout.
2. Acceder a UI strings vía `site.data.i18n[page.lang].xxx`.
3. Acceder a campos traducibles con lógica condicional. Patrón Liquid:

   ```liquid
   {% if page.lang == "en" %}
     {{ restaurante.por_que_kids_friendly_en }}
   {% else %}
     {{ restaurante.por_que_kids_friendly }}
   {% endif %}
   ```

4. En `<head>` de cada página añadir `hreflang`:

   ```html
   <link rel="alternate" hreflang="es" href="https://conparquedebolas.com/restaurantes/{{ slug }}/">
   <link rel="alternate" hreflang="en" href="https://conparquedebolas.com/en/restaurantes/{{ slug }}/">
   <link rel="alternate" hreflang="x-default" href="https://conparquedebolas.com/restaurantes/{{ slug }}/">
   ```

   El `hreflang="en"` solo se incluye si la versión EN existe (`page.has_en == true` en páginas ES, o siempre en páginas EN).

5. `<title>` y `<meta name="description">` deben generarse a partir de strings traducidas según `page.lang`.

6. La etiqueta `<html>` debe llevar el atributo `lang`:

   ```html
   <html lang="{{ page.lang }}">
   ```

   **Importante:** este atributo está en `_layouts/default.html` de la gema Minima. Para modificarlo hay que copiar ese fichero al directorio `_layouts/` del proyecto y mantenerlo localmente. Esto desvincula las actualizaciones automáticas del tema para ese fichero — aceptable dado que el cambio es mínimo y estable.

---

## Switcher de idioma

Añadir en el header de todos los layouts y en `index.html` un enlace que apunte a la versión equivalente en el otro idioma.

Reglas:
- En cualquier página ES, el enlace lleva al mismo slug bajo `/en/`.
- En cualquier página EN, el enlace quita el prefijo `/en/`.
- El switcher solo se muestra si `page.has_en == true` (en páginas ES) o siempre (en páginas EN, porque la ES siempre existe).
- El texto del enlace viene de `site.data.i18n[page.lang].switch_lang`.

---

## `index.html`

La home también es bilingüe. Generar dos versiones:
- `/index.html` en español (existente).
- `/en/index.html` en inglés, listando solo restaurantes con traducción EN.

La versión EN de la home se genera desde el plugin (igual que las páginas de ciudad y filtro), no como fichero estático, para poder filtrar los restaurantes por disponibilidad de traducción.

---

## Reglas duras (no romper)

1. **Nunca** publicar páginas EN con texto en español mezclado. Si falta traducción → omitir página.
2. **Nunca** traducir nombres propios: restaurante, ciudad, calle, dirección.
3. Mantener un único YAML por provincia. No duplicar ficheros por idioma.
4. No usar widgets de traducción automática (Google Translate, etc.). Todo el contenido es pre-traducido y estático.
5. Los `slug` siempre en español, en ambos idiomas — incluyendo los slugs de tags.
6. Toda string visible en la web debe pasar por `_data/i18n.yml`. Prohibido string literal en layouts.

---

## Reglas para nuevas entradas en el YAML

Cuando se añada un restaurante:

1. Rellenar todos los campos en español como hasta ahora (ver `CLAUDE.md`).
2. Añadir las variantes inglesas: `por_que_kids_friendly_en`, `descripcion_general_en`.
3. Si la traducción inglesa no está lista al momento de añadir el restaurante: **omitir** los campos `_en` (no dejarlos como `""`). El plugin trata "ausente" como "no publicar en inglés".
5. Cuando la traducción esté lista, añadir los campos `_en` en un commit posterior.

---

## SEO

- `hreflang` obligatorio en cada página (ver sección de layouts).
- `<title>` y `<meta description>` traducidos en cada idioma.
- `sitemap.xml` debe incluir ambas versiones de cada página. El plugin `jekyll-sitemap` auto-incluye todas las páginas generadas; verificar que está en el `Gemfile`.
- `<html lang="...">` correcto en cada página.

## Idiomas futuros (FR, DE) — solo si los analytics lo justifican

Si en algún momento se decide ampliar:

- Replicar el patrón añadiendo sufijos `_fr`, `_de` a los campos traducibles del YAML.
- Añadir entradas `fr:`, `de:` en `_data/i18n.yml`.
- Añadir prefijos `/fr/`, `/de/` en el plugin con la misma regla de "omitir si falta traducción".
- No requiere refactor estructural.

Si en el futuro se superan 3 idiomas, considerar refactor del YAML al patrón anidado:

```yaml
por_que_kids_friendly:
  es: "..."
  en: "..."
  fr: "..."
```

Esto no es necesario hoy.
